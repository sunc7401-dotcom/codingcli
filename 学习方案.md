## 0. 项目定位先说清楚

当前仓库不是单一的 Java 重构工具，而是一个 Python 版终端 Agent 框架，里面包含：

- 通用 PaiCLI Agent：对话、工具调用、记忆、RAG、MCP、浏览器、技能、渲染、HITL 审批等基础设施。
- Java 代码检查与安全重构 Agent：`suncli_py/refactor_agent`，这是你简历里最适合重点包装的 AI 应用项目。

简历主线建议聚焦第二部分：

> 面向 Java Maven 仓库的代码检查与安全重构 Agent。系统先用 JavaParser AST、Symbol Solver、静态规则和测试反馈生成结构化事实，再由 LLM 作为决策核心完成 issue triage、重构计划、受控 edit operations 和失败修复 loop，最后通过 patcher、AST validator、Maven 编译测试、JaCoCo 覆盖感知、任务快照和回滚机制保证落地安全。

当前代码的关键特点：

- LLM 是 refactor-agent 的必要组件。`scan`、`plan`、`apply` 都会通过 `_require_llm_assistant()` 强制要求 LLM 配置。
- AST、Symbol Solver、规则扫描不是最终决策者，而是给 LLM 提供候选问题和证据。
- LLM 不直接写文件。LLM 只能返回结构化 JSON，真正写文件的是 `RefactorPatcher`。
- LLM 可以调用只读工具获取更多上下文。
- apply 后可以触发 verify，验证失败时可以进入 LLM repair loop。
- 当前 refactor-agent 还没有真正接入主 Agent 的长期记忆系统，这是后续非常适合升级的点。

### 3.2 代码逻辑

`models.py` 是业务协议层。扫描、计划、patch、验证、回滚之间不传散乱字典，而是传结构化 dataclass。

最重要的几个对象：

- `ProjectProfile`：项目画像，记录是否是 Git、是否是 Maven、是否有源码/测试目录、Java/Maven 版本、多模块信息和 warnings。
- `RefactorIssue`：一个候选或确认后的代码问题，包括类型、文件位置、证据、影响、建议、风险、是否可自动应用。
- `Evidence`：问题来源证据，例如 AST 行号、Symbol Solver 解析结果、规则阈值、LLM triage 决策。
- `JavaContext`：给 LLM 和计划阶段使用的上下文，包括源码片段、相关测试、直接调用方、warnings。
- `RefactorPlan`：可审查的重构计划，包括修改文件边界、预期改动、不做什么、验证命令、回滚策略。
- `VerificationResult`：编译、测试、覆盖率和静态检查的统一结果。
- `RollbackResult`：回滚是否成功、恢复了哪些文件、是否存在冲突。

### 3.3 为什么这样设计

这种模型层设计有四个好处：

1. 阶段边界清晰。`scan` 产出 `ScanResult`，`plan` 产出 `RefactorPlan`，`verify` 产出 `VerificationResult`。
2. 方便持久化。核心对象都有 `to_dict/from_dict`，可以写入 `.paicli/refactor-agent`。
3. 方便测试。单元测试可以直接构造 `RefactorIssue`、`RefactorPlan`，不用真的跑完整 CLI。
4. 方便约束 LLM。LLM 输出的内容会被映射回枚举和 dataclass，不是直接相信自然语言。

### 3.4 如何测评

重点测这些：

- `to_dict/from_dict` 是否能往返。
- 老 JSON 缺字段时是否有合理默认值。
- `RefactorIssue.file_path/start_line/end_line` 是否能准确定位源码。
- `RefactorPlan.files_to_modify` 是否能限制 patch 范围。
- `VerificationResult.status` 是否能表达 passed / warning / failed。

### 4.2 代码逻辑

`ProjectDetector.detect()` 在真正扫描前做运行环境画像：

- 当前目录是不是 Git 仓库。
- 当前目录是不是 Maven 项目。
- 是否存在 `src/main/java`。
- 是否存在 `src/test/java`。
- Git 工作区是否干净。
- Maven 版本。
- Java 版本。
- Maven 多模块结构。

`commands.run_scan()` 会强依赖这个结果：

- 非 Git 仓库会抛 `RefactorAgentError`。

### 4.3 为什么这样设计

重构 Agent 不能脱离工程环境运行。后续环节需要：

- Git 信息用于快照、风险提示和回滚证据。
- Maven 用于编译、测试、JaCoCo 和 JavaParser helper 执行。
- Java 源码目录用于 AST 扫描。
- 测试目录用于覆盖感知和 characterization test。
- 多模块信息为后续 affected test selection 留接口。

提前失败比中途失败更安全。

### 4.4 如何测评

构造临时目录：

- 有 `.git`、`pom.xml`、`src/main/java` 时应该识别为合法项目。
- 没有 `pom.xml` 时 `scan` 应该报错。
- Git dirty 时应该出 warning 或交互确认。
- 多模块 `pom.xml` 应该能识别 module。

### 5.2 代码逻辑

Python 不能直接使用 JavaParser 的 JVM 对象，所以这里采用“Python 编排 + Java helper 输出 JSON”的方式。

执行流程：

```text
JavaSmellScanner.scan
  -> JavaParserAnalyzer.analyze_files(paths)
  -> mvn -q -f java_ast_helper/pom.xml compile exec:java
  -> JavaAstDump.main(root, relative_java_files...)
  -> StaticJavaParser.parse(source)
  -> Symbol Solver 解析方法、字段、调用归属
  -> 输出 JSON
  -> Python _ast_file_from_dict 转成 dataclass
```

`JavaParserAnalyzer.analyze_files()` 在 `suncli_py/refactor_agent/java_ast.py:97` 构造 Maven 命令，运行 helper。失败时抛 `JavaAstError`。当前 scanner 要求 AST 成功，缺失 AST 会在 `scanner.py:148` 抛错。

Java helper 做了几件关键事：

- 找出 class、enum、record。
- 找出 method 和 constructor。
- 提取方法起止行、签名、声明类型、是否 private、是否 static。
- 提取 method call 的名称、行号、scope、declaring type、resolved signature、return type。
- 提取 field access 的名称、行号、scope、declaring type、字段类型。
- 通过 `CombinedTypeSolver` 注册 JDK 类型和项目里的 `src/main/java`、`src/test/java`。

### 5.3 为什么这样设计

如果只用正则或文本扫描 Java，会有很多误判：

- 方法重载无法区分。
- 注释、字符串、局部变量名会干扰识别。
- private 方法是否真的没人调用，不能只数方法名。
- Feature Envy 需要知道访问的是哪个类的成员。
- 影响范围分析需要知道谁调用了目标方法。

JavaParser 解决“语法结构”问题，Symbol Solver 解决“这个调用到底指向谁”的问题。

### 5.4 好处

- Dead Code 可以基于 `resolved_signature` 判断调用关系。
- Feature Envy 可以统计外部类型成员访问。
- LLM 拿到的是结构化事实，而不是靠猜。
- patch validator 可以复用 AST 重新解析修改后的代码。

### 5.5 当前边界

当前 Symbol Solver 主要覆盖源码根：

- `src/main/java`
- `src/test/java`

复杂场景仍可能解析失败：

- 多模块依赖关系很复杂。
- Spring 动态代理、反射、注解生成调用。
- Lombok 生成代码。
- 第三方依赖源码不可见。
- 泛型和重载特别复杂。

当前实现会用 `symbol_resolved=false` 或 `error` 字段保存失败信息。后续可以扩展 Maven classpath、依赖 jar、Lombok delombok 或 Spoon/OpenRewrite。

### 5.6 如何测评

对应测试：

- `tests/test_refactor_agent_java_ast.py:15`
- `tests/test_refactor_agent_java_ast.py:36`
- `tests/test_refactor_agent_java_ast.py:52`

测评指标：

- 方法行号是否准确。

### 6.2 代码逻辑

`JavaSmellScanner.scan()` 做的是“候选粗筛”，不是最终决策。

流程：

```text
收集 Java 文件
  -> JavaParserAnalyzer 分析所有文件
  -> AST 分析结果转 JavaFileAnalysis
  -> 执行各类规则
  -> 生成 RefactorIssue 候选
  -> 按文件、行号、类型排序并分配 RA-0001 这类 id
```

当前候选类型：

- Long Method
- Large Class
- Complex Condition
- Unclear Naming
- Dead Code
- Feature Envy
- Duplicate Code

重点看两个体现 AST/Symbol Solver 价值的规则。

#### Dead Code

对应 `scanner.py:410`。

逻辑：

- 只关注 private 方法，跳过 `main`。
- 收集所有已解析的方法调用 `resolved_signature`。
- 如果 private 方法有 resolved signature 且出现在调用集合里，就不是 dead code。
- 如果没有符号信息，则用 identifier count 做兜底。
- 生成 `REMOVE_DEAD_CODE` 建议，通常低风险、可自动应用。

#### Feature Envy

对应 `scanner.py:459`。

逻辑：

- 跳过 private 方法和无法解析符号的方法。
- 统计方法内部访问的外部方法调用和外部字段。
- 排除本类/内部类和 `java.*` 类型。
- 要求外部访问数量达到阈值，并且显著超过本地访问。
- 找到 dominant external type 后生成 `MOVE_METHOD` 建议。

#### Duplicate Code

对应 `scanner.py:535`。

逻辑：

- 优先执行 `mvn -q pmd:cpd-check`。
- 如果 CPD 不可用或输出不可解析，降级到本地滑动窗口规则。
- 本地规则会对代码行做 normalization，再找重复窗口。

### 6.3 为什么这样设计

当前扫描器的定位是“给 LLM 提供候选和证据”，不是“单独决定问题一定成立”。

这样设计有三个原因：

1. 静态规则适合快速粗筛，成本低、可重复、可测试。
2. AST/Symbol Solver 提供可信事实，避免纯文本误判。
3. LLM 负责理解上下文、判断优先级、解释影响、决定是否适合自动修。

这就是“LLM as decision layer over deterministic facts”。

### 6.4 当前需要注意的细节

`llm_assistant.triage_issues()` 会要求 LLM 返回每个候选的决策，但当前实现里，如果 LLM 没有返回某个 issue 的 decision，它会把原始 candidate 继续保留，只是排序靠后。代码在 `llm_assistant.py:72` 附近。

这说明当前已经强制走 LLM 分支，但“LLM 显式拒绝候选”的能力还可以继续增强。更理想的 schema 应该包含：

```json
{
  "id": "RA-0001",
  "is_real_issue": true,
  "decision": "keep|drop|needs_more_context"
}
```

然后 scan 阶段只保存 LLM keep 的 issue。

### 7.7 _chat_json 和工具调用

对应 `llm_assistant.py:251`。

流程：

```text
messages = [system, user]
schemas = tools.schemas()
循环最多 max_tool_rounds + 1 次：
  response = client.chat(messages, tools=schemas)
  if response.has_tool_calls():
      把 assistant tool_call 消息加入 messages
      执行 tools.execute(name, args)
      把 tool result 加入 messages
      continue
  else:
      解析 response.content 里的 JSON
```

这让 LLM 在决策前可以主动调用工具补上下文，例如：

- 不确定源码片段时调用 `read_file`
- 不确定调用关系时调用 `search_code`
- 不确定 issue 信息时调用 `get_issue_context`
- repair 时调用 `get_verification_feedback`

### 7.8 为什么这样设计

成熟 code agent 不应该是“prompt 一把梭改文件”。更可靠的结构是：

```text
LLM 决策核心
  -> 工具获取事实
  -> JSON 结构化输出
  -> 确定性代码校验输出
  -> 确定性 patcher 落地
  -> 编译测试反馈
  -> LLM repair loop
```

好处：

- 可控：模型不能越权直接写文件。
- 可审计：每次输出是 JSON，可以保存。
- 可验证：落地前后都有 deterministic checks。
- 可迭代：验证失败能把错误反馈喂回模型。
- 好展示：面试时能说明“LLM 负责决策，系统负责验证和执行”。

### 8.2 代码逻辑

`BASE_AGENT_RULES` 定义所有阶段通用约束：

- AST facts、Symbol Solver facts、规则 findings、测试和工具输出都是 evidence。
- 不要因为规则报了 candidate 就直接认为是真问题。
- 证据不足时先调用 read/search/context 工具。
- 不直接修改文件，只返回 JSON 决策或 edit operations。
- 不触碰 allowed_files 之外的文件。
- 除非计划允许，否则保留 public API。
- 优先最小行为保持变更。
- 最终回答必须是符合 schema 的 JSON，不返回 Markdown。

各阶段 prompt 再补充阶段目标：

- triage：判断候选是否真实、优先级、风险、自动化适用性。
- planning：生成用户可审查计划，不生成代码。
- edit：生成受控 edit operations。
- repair：根据验证失败反馈生成最小修复。

### 8.3 为什么这样设计

系统提示词是“业务流程约束”。如果没有这些规则，模型可能会：

- 跳过工具，凭感觉判断。
- 直接返回自然语言。
- 修改计划外文件。
- 改 public API。
- 在 planning 阶段提前生成代码。
- verify 失败时扩大修改范围。

把这些约束集中在 `prompts.py`，可以让各阶段 LLM 行为一致，也方便后续迭代。

### 8.4 如何测评

可测点：

- mock LLM 返回 Markdown 是否能被 `_parse_json_object` 提取 JSON。
- LLM 尝试修改 allowed_files 外文件，patcher 是否拦截。
- planning 阶段是否只生成计划，不生成 edits。
- repair 阶段是否能读取 verification feedback。
- prompt 修改后现有 LLM fake tests 是否仍通过。

### 8.5 简历写法

> 为代码重构 Agent 设计分阶段系统提示词，将 triage、planning、edit、repair 拆分为不同 LLM 角色，并通过 JSON schema、allowed_files、public API 保持、工具优先和最小变更原则约束模型行为，提升自动化代码修改的可控性和可审计性。

### 12.2 LLM 建议怎么被识别并修改代码

LLM 通过 `generate_edit_plan()` 返回 JSON：

```json
{
  "edits": [
    {
      "file_path": "src/main/java/demo/UserService.java",
      "start_line": 12,
      "end_line": 18,
      "replacement": "新的代码文本"
    }
  ],
  "explanation": "..."
}
```

`RefactorPatcher._changes_from_llm_edits()` 读取这些 edits：

1. 遍历 `edit_plan["edits"]`。
2. 取出 `file_path/start_line/end_line/replacement`。
3. 检查 `file_path` 是否在 `plan.files_to_modify` 里。
4. 通过 `_resolve_allowed_file()` 确认路径没有越界、不是 ignored 目录、文件存在。
5. 读取原文件。
6. 按行号替换指定范围。
7. 生成 `PatchChange(before_text, after_text)`。

真正写文件发生在 `apply_changes()`：

1. 写 snapshot。
2. 生成 unified diff。
3. 写 `patch.diff`。
4. 把 `after_text` 写入真实文件。
5. 重新读取文件确认写入结果。
6. 调用 `AstPatchValidator.validate()`。
7. 如果失败，恢复 before_text。
8. 成功后写 after 状态和 diff_summary。

### 12.3 为什么 LLM 不能直接写文件

“LLM 不能直接写文件，LLM 只能生成 edit operations，RefactorPatcher 负责落地”的意思是：

- LLM 只描述想怎么改。
- patcher 决定这个改动能不能执行。
- patcher 才拥有最终文件写入权。

原因：

- 可以限制修改范围：只能改 `plan.files_to_modify`。
- 可以检查路径越界：不能写 `.git`、`target`、项目外文件。
- 可以检查行号合法性。
- 可以生成 diff 和 snapshot。
- 可以在 AST 校验失败时自动恢复。
- 可以把模型输出从“不可控自然语言”变成“可验证操作”。

这就是安全代码 Agent 的关键设计。

### 12.4 当前仍保留的 deterministic patch

`generate_changes()` 里还有两个非 LLM patch 分支：

- Dead Code 的 `_remove_dead_code()`。
- Long Method 的 `_extract_accumulator_method()`。

但当前 `commands.run_apply()` 必须先拿到 `llm_edit_plan`，没有 LLM edit plan 会报错。所以业务主路径已经是 LLM 生成修改方案。保留这些 deterministic 方法更像备用能力和历史兼容代码。

### 13.1 对应源码位置

### 13.2 代码逻辑

patch 写入后，validator 会重新分析 Java AST。

核心检查：

- 修改后的 Java 文件能否被 JavaParser 正常解析。
- class 声明是否异常变化。
- externally visible signatures 是否异常变化。

- 有些 public API 变化可能当前测试没覆盖。
- 有些语法结构问题可以在更早阶段发现。
- LLM 可能为了修局部问题改掉方法签名。

AST patch validator 是“结构安全阀”。

### 13.5 简历写法

> 实现 AST 级 patch 校验器，在 LLM patch 写入后重新解析 Java 文件并比较类声明和非 private 方法签名，拦截 public API 意外变化和语法结构破坏，作为编译测试前的结构安全防线。

### 16.4 为什么不用 git reset --hard

`git reset --hard` 会破坏用户工作区里所有未提交改动。当前系统只恢复本次 task 涉及的计划文件。

1. 读取 snapshot。
2. 检查当前文件 sha 是否等于 apply 后 sha。
3. 如果不等，说明 apply 后用户又改过该文件，返回 conflict。
4. 用户确认 force 后才恢复。
5. 只恢复 before 目录里的任务文件。

这比全局重置安全得多。

### 18.3 refactor-agent 当前有没有用记忆系统

没有真正接入。

我在源码中看到：

- `suncli_py/refactor_agent` 没有引用 `MemoryManager`。
- refactor-agent 自己用 `RefactorAgentToolbox` 提供上下文工具。
- refactor-agent 自己用 `prompts.py` 提供阶段提示词。
- refactor-agent 不会自动读取长期记忆或 `PAI.md`。

所以当前：

```text
refactor-agent 有上下文系统
但没有跨任务记忆系统
```

### 18.4 为什么这是一个好升级点

如果把记忆接进去，可以让 Agent 记住：

- 用户偏好：保守重构、不要改 public API、先生成测试。
- 项目约定：Controller 不访问 Repository、Service 命名规范。
- 历史失败：某模块测试脆弱、某类 patch 容易编译失败。
- 历史决策：某个 issue 被用户判定为误报。
- 修复经验：某类 verify failure 应该如何 repair。

建议升级方式：

```text
run_scan / run_plan / run_apply
  -> 创建或注入 MemoryManager
  -> 根据 issue/file/project 检索相关记忆
  -> 注入 LLM prompt payload
  -> 增加 memory_search / memory_save 工具
  -> 在 report 后保存关键决策和失败经验
```

### 18.5 简历写法

如果当前版本不接记忆，不要夸大。可以写：

> 项目已有独立记忆系统，支持短期会话、长期事实、项目 PAI.md 和工具结果记忆；重构 Agent 当前主要使用本轮工具上下文，后续规划将用户偏好、项目规范、历史失败和重构经验接入 triage、plan、repair 决策。

如果你后续实现接入后，可以改成：

> 将项目级长期记忆接入 Java 重构 Agent，在 triage、planning、edit 和 repair 阶段检索用户偏好、项目约定和历史失败经验，并允许 LLM 通过 memory 工具保存重构决策，使 Agent 具备跨任务学习和项目适配能力。

## 21. 当前项目成熟度与不足

### 21.1 已经比较成熟的点

- 有完整 CLI 工作流：scan、plan、apply、verify、characterize、rollback、report。
- 用 JavaParser AST 作为 Java 结构事实，不是只靠文本。
- 用 Symbol Solver 解析方法调用、字段访问和类型归属。
- LLM 是 scan/plan/apply 的必要组件。
- LLM 可以调用只读工具补上下文。
- LLM 输出 JSON edit operations，不直接写文件。
- patcher 做 allowed_files 和路径边界检查。
- AST patch validator 拦截结构风险。
- Maven compile/test/JaCoCo 提供反馈。
- verify failed 后有 LLM repair loop。
- 有任务级 snapshot 和 rollback。
- 测试覆盖了核心风险场景。

### 21.2 仍然是 MVP 或需要优化的点

- refactor-agent 尚未接入主记忆系统。
- triage schema 还缺少显式 `keep/drop`，LLM 未返回的候选仍会被保留。
- `search_code()` 是字面量搜索，不是语义检索。
- Symbol Solver 对多模块、Lombok、Spring 动态行为支持有限。
- `generate_edit_plan()` 使用 `Path(".").resolve()`，最好显式传 root，避免调用环境变化。
- JSON schema 校验较轻，缺少严格 schema validator。
- repair loop 的最大轮次和失败分类还比较简单。
- deterministic patch 分支仍存在，和“LLM 生成修改”的产品叙事需要进一步统一。
- 部分中文输出字符串存在编码乱码，影响 CLI 展示和文档可读性。
- 没有系统 benchmark 和指标统计脚本。

### 21.3 优先升级建议

优先级从高到低：

1. triage schema 增加 `is_real_issue` / `decision=keep|drop|needs_more_context`，scan 只保存 LLM keep 的 issue。
2. 将 `MemoryManager` 接入 refactor-agent，注入用户偏好、项目约定、历史失败。
3. 增加严格 JSON Schema 校验和自动重试。
4. 增加 AST symbol 工具，例如 find_callers、find_references、find_related_tests。
5. 使用 RAG/embedding 或 ripgrep index 提升 code search。
6. 接入 Spoon/OpenRewrite 执行更复杂 AST 级重构。
7. 建立 benchmark，统计 precision、patch success、repair success。
8. 清理源码中文乱码。

## 22. 简历怎么写

### 22.1 项目名称

可选：

- Java 安全重构 Agent
- AI Code Review & Safe Refactor Agent
- 基于 AST 与 LLM 的 Java 代码检查和安全重构系统
- Java Repository Refactoring Agent

### 22.2 项目描述

推荐版本：

> 设计并实现面向 Java Maven 仓库的代码检查与安全重构 Agent，结合 JavaParser AST、Symbol Solver、静态规则和 LLM 决策能力识别 Long Method、Dead Code、Feature Envy、Duplicate Code 等代码坏味道，并生成可审查的小步重构计划。系统支持 LLM 工具调用、JSON edit operations、用户确认、受控 patch 落地、AST Patch Validator、Maven 编译测试、JaCoCo 覆盖感知、验证失败 repair loop、任务级快照回滚和 Markdown 证据报告，降低 LLM 自动修改代码引入回归的风险。

### 22.3 技术亮点 bullet

可以写 5 条：

1. 基于 JavaParser 和 Symbol Solver 构建 Java 语义分析层，提取类、方法、方法调用、字段访问、声明类型和 resolved signature，为坏味道识别和 LLM 决策提供结构化代码事实。
2. 设计 LLM 决策核心，将静态规则结果作为候选 evidence，由 LLM 完成 issue triage、优先级排序、风险判断、重构策略选择和用户可审查计划生成。
3. 封装只读工具调用体系，向 LLM 暴露 issue context、文件片段读取、代码搜索、计划上下文和验证反馈，使模型能按需检索上下文而不是盲目扫描全仓。
4. 实现受控 patch 机制，LLM 仅输出 JSON edit operations，系统通过 allowed_files、路径边界、行号范围、AST 结构校验和 public API 签名检查决定是否落地。
5. 构建验证与反馈闭环，自动执行 Maven 编译、测试和 JaCoCo 覆盖分析，失败时回滚到任务快照并将错误反馈交给 LLM 生成 repair edits，形成可审计、可恢复的代码 Agent 工作流。

### 22.4 面试怎么讲“LLM 是核心，不是摆设”

可以这样说：

> 这个项目不是只让 LLM 写解释。当前 scan、plan、apply 都强制要求 LLM。静态规则和 AST 只产出候选事实，LLM 在 triage 阶段判断问题是否值得修、优先级、风险和重构策略；plan 阶段生成用户可审查计划；apply 阶段生成 JSON edit operations；verify 失败后进入 repair loop，由 LLM 基于测试错误和 diff 生成修正版 edits。工具和规则是辅助模型决策的事实来源，patcher 和 verifier 是安全执行与反馈系统。

### 22.5 面试怎么讲“为什么不让 LLM 直接改文件”

可以这样说：

> 我把 LLM 和文件写入权隔离开。LLM 只能返回结构化 edits，例如 file_path、start_line、end_line、replacement。真正写文件的是 RefactorPatcher，它会检查文件是否在 confirmed plan 的 allowed_files 里、路径是否越界、行号是否合法，并在写入后用 AST Patch Validator 检查 Java 结构和 public API 签名。如果任何一步失败，系统恢复 before snapshot。这样保留了 LLM 的代码生成能力，同时用确定性系统控制风险。

### 22.6 面试怎么讲“工具调用”

可以这样说：

> LLM 决策前不直接猜全仓库，而是通过工具获取上下文。系统向模型暴露 get_issue_context、read_file、search_code、get_plan_context、get_verification_feedback 等只读工具。比如 triage 时模型可以查看源码片段、相关测试和调用方；repair 时可以读取验证失败信息和当前文件内容。工具没有写权限，写文件只能经过 patcher。

### 22.7 面试怎么讲“闭环 Agent”

可以这样说：

> 我把重构流程做成闭环：LLM 生成 edits 后，系统 apply patch，随后运行 AST 校验、Maven 编译、单元测试和覆盖率分析。如果失败，系统先回滚到任务快照，再把错误摘要、verification result、当前文件片段和原计划作为工具上下文交给 LLM，生成 revised edit operations，并限制最多迭代 N 次。这让 Agent 不只是一次性生成代码，而是能根据环境反馈继续修正。

## 24. 一句话总结

这个项目最值得讲的不是“写了几个代码坏味道规则”，而是：

> 用 JavaParser AST、Symbol Solver、静态规则和测试反馈构建可验证证据系统，让 LLM 作为代码检查与重构决策核心，通过工具按需获取上下文，生成可审查计划和受控 edit operations，再由 patcher、AST validator、Maven 验证、repair loop、快照回滚和报告系统保证自动化重构安全落地。
