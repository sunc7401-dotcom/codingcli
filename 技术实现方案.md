# Java 代码审查与自动化安全重构 CLI 系统技术实现方案

## 1. 方案结论

本项目推荐在当前 `paicli-py` 项目基础上改造实现，而不是直接二次开发通用开源 Coding Agent，也不建议从零编写。

当前项目已经具备本需求所需的大量通用底座能力，包括本地 CLI、LLM 客户端抽象、工具注册、HITL 人工确认、路径与命令安全策略、审计日志、快照、代码检索、RAG、Java 文件搜索与分块等能力。这些能力与需求中的“本地运行、用户确认、可验证、可回滚、模型无关接入”高度匹配。

但当前项目的 Java 能力主要停留在文件搜索、正则分块和通用 Agent 工具调用层，尚不足以支撑“AST 级 Java Maven 安全重构”。因此技术路线应是：复用当前 CLI/Agent/安全/快照/LLM 底座，在其上新增一个独立的 `refactor-agent` 业务域，并引入 JavaParser、OpenRewrite、Spoon、PMD CPD、JaCoCo 等 Java 生态工具，形成专门面向 Java Maven 仓库的小步重构闭环。

最终 MVP 形态为独立命令：

```bash
refactor-agent scan
refactor-agent plan --issue <id>
refactor-agent characterize --issue <id>
refactor-agent apply --issue <id>
refactor-agent verify
refactor-agent rollback
refactor-agent report
```

## 2. 调研依据与可行性判断

### 2.1 代码审查与 Coding Agent 产品边界

GitHub Copilot Code Review、OpenAI Codex GitHub Review、Claude Code Review、Qodo、CodeRabbit 等产品主要面向 Pull Request 审查。OpenAI Codex GitHub Review 官方说明其审查 PR diff、遵循仓库指导并发布聚焦严重问题的 GitHub code review。Claude Code Review 官方说明其通过多 Agent 分析完整代码库上下文，并在 GitHub PR 中发布内联发现。这类产品适合作为竞品参考，但不适合作为本项目的直接技术底座，因为需求重点是本地 Maven 仓库中的小步重构、确认后修改、失败回滚与证据链记录。

Aider、SWE-agent、OpenHands 等开源 Coding Agent 更偏通用软件工程任务。Aider 官方定位是终端 AI 结对编程，可编辑本地 Git 仓库并连接多种 LLM；SWE-agent 和 OpenHands 更强调通用 agentic coding、自动修复 issue、运行工具与测试。它们适合作为 Agent 交互与工程经验参考，但本需求需要更强的 Java AST、安全重构、Maven 验证和覆盖感知能力，不宜直接二改。

### 2.2 Java 分析与重构工具可行性

OpenRewrite 官方 Maven 插件支持 `rewrite:dryRun` 生成预览 patch，也支持 `rewrite:run` 将 recipe 变更应用到本地源码，适合作为可预览、可审查、可回滚的 Java 自动重构执行器。

JavaParser 能够直接解析 Java 源码并生成真实 AST，提供类、接口、枚举、record、构造器、方法、字段、修饰符、源码区间等结构信息。JavaParser Symbol Solver 进一步提供方法调用解析、字段引用解析、声明类型和 resolved signature，用于提升 Dead Code、Feature Envy、调用影响范围和测试选择的可信度。对于 Python 编排层，推荐通过一个轻量 Java helper 调用 JavaParser，将 AST 和符号摘要以 JSON 返回给 `refactor-agent`，用于替代正则方法/类提取，并作为坏味道识别、上下文收集、低风险补丁范围校验和后续 AST 级变换的基础。

Spoon 官方定位是 Java 源码分析、重写、转换和转译库，能够构建 Java AST，并提供分析和转换 API，适合用于自定义坏味道检测、上下文提取、风险判断和部分 AST 级变换。

PMD CPD 官方支持 Java 重复代码检测，并可通过 Maven PMD Plugin 的 `cpd-check` goal 运行，适合覆盖 MVP 中“重复代码”坏味道识别。

JaCoCo Maven Plugin 可在 Maven 测试执行过程中生成覆盖率报告，适合做测试覆盖感知，判断修改类、方法或行是否被测试触达。

因此，本项目技术可行，关键在于将 LLM 约束在“解释、计划、补丁建议和上下文理解”位置，把代码修改落到可解析、可审查、可验证的 patch 或 AST/recipe 执行链路中。

## 3. 建设目标

### 3.1 产品目标

实现一个面向个人开发者和小团队的本地 Java Maven 安全重构 CLI 工具。工具能够扫描真实 Java Maven Git 仓库，识别代码坏味道，生成可解释的小步重构计划，在用户确认后应用修改，并通过 diff、编译、测试、静态分析、覆盖感知和快照回滚控制风险。

### 3.2 工程目标

- 复用当前 `paicli-py` 的 CLI、LLM、HITL、安全策略、审计、快照和报告能力。
- 新增独立 `refactor-agent` 命令入口，避免污染原有交互式 `paicli` 工作流。
- Java 分析与重构能力独立成业务模块，便于后续扩展 Gradle、PR Bot、IDE 插件和更多语言。
- LLM 接入必须模型无关，明确支持 OpenAI、Claude、DeepSeek。
- 所有写文件动作都必须由用户确认后发生。
- 一次只做一个小步重构任务，且必须保留证据链。

## 6. 核心模块设计

### 6.1 CLI 命令模块

模块：`suncli_py.refactor_agent.cli`

职责：

- 解析 `refactor-agent` 子命令。
- 加载配置。
- 初始化 LLM、项目路径、安全策略和本地状态。
- 调用对应 command handler。
- 输出用户可读结果。

命令：

```bash
refactor-agent scan [--format text|json] [--output <path>]
refactor-agent plan --issue <id>
refactor-agent apply --issue <id> [--yes]
refactor-agent characterize --issue <id>
refactor-agent verify [--issue <id>]
refactor-agent rollback [--task <id>]
refactor-agent report [--latest|--task <id>]
```

MVP 中 `--yes` 只能跳过二次确认，不能跳过高风险确认；高风险任务仍必须交互确认。

### 6.2 项目检测模块

模块：`project_detector.py`

检测项：

- 是否存在 `.git`。
- 是否存在 `pom.xml`。
- 是否存在 `src/main/java`。
- 是否存在 `src/test/java`。
- 是否可执行 `mvn -v`。
- 当前 Git 工作区是否干净。
- 是否存在多模块 Maven 结构。
- 是否存在测试框架迹象，如 JUnit、TestNG、Mockito。

输出模型：

```python
ProjectProfile:
  root: Path
  is_git_repo: bool
  is_maven_project: bool
  has_main_java: bool
  has_test_java: bool
  is_git_clean: bool
  maven_version: str | None
  java_version: str | None
  modules: list[MavenModule]
  warnings: list[str]
```

若工作区不干净，`scan` 可以继续，但 `apply` 前必须确认，并在快照中记录修改前已有变更，确保 rollback 不覆盖用户原有未提交修改。

### 6.3 坏味道扫描模块

模块：`scanner.py`、`java_ast.py`、`java_ast_helper/` 与 `smell_rules/`

扫描分两层：

1. JavaParser AST + Symbol Solver 扫描：通过本地 Maven helper 调用 JavaParser，解析真实 Java AST，提取类、接口、枚举、record、构造器、方法、源码起止行、修饰符、签名、声明类型、resolved signature、方法调用和字段访问，作为坏味道识别的优先事实来源。
2. 快速本地降级规则扫描：当 JavaParser helper 不可用、Maven 不可执行或源码存在语法错误时，降级到文本启发式规则，保证基础结果稳定。
3. LLM 辅助解释与建议：只对候选问题补充说明、风险和重构建议，不作为唯一事实来源。

JavaParser AST / Symbol 输出要求：

- Python 编排层不得直接依赖 JVM 内存对象，应通过 helper 输出结构化 JSON。
- AST 摘要只包含文件路径、结构类型、符号名、源码范围、签名、修饰符、声明类型、resolved signature、方法调用和字段访问等必要信息，不上传完整仓库。
- `scanner.py` 应优先消费 JavaParser AST 结果；仅在 AST 解析失败时写入 warning 并降级。
- Dead Code、Feature Envy 和影响范围分析应优先消费 Symbol Solver 结果；解析失败或 `resolved=false` 时降级到原有文本规则。
- 后续低风险补丁应用应优先使用 AST 行号范围校验，避免正则误删注释、字符串或相邻方法。

MVP 支持 7 类坏味道。

#### 长方法

检测依据：

- 方法行数超过阈值，如 80 行。
- 分支数量超过阈值，如 `if/else/switch/for/while/catch` 总数超过 12。
- 嵌套深度超过阈值，如 4。

推荐重构：

- Extract Method。
- 拆分条件判断。
- 提取局部业务步骤。

自动化等级：

- 方法内部存在清晰连续代码块且无复杂控制流穿越时，可自动。
- 涉及跨分支状态、副作用和异常流程时，只生成计划。

#### 重复代码

检测依据：

- PMD CPD 输出。
- 相似代码块行数和 token 数超过阈值。

推荐重构：

- Replace Duplicate Logic With Shared Method。
- Extract Method。

自动化等级：

- 同类内部重复可自动尝试。
- 跨类、跨模块重复默认只生成计划。

#### 过大类

检测依据：

- 字段数、方法数、总代码行数超过阈值。
- 公开方法过多。
- 类内方法访问字段呈现多个聚类。

推荐重构：

- Extract Class。
- Move Method。

自动化等级：

- MVP 默认不自动执行 Extract Class，只生成计划和风险说明。

#### 复杂条件

检测依据：

- 嵌套 `if/else` 深度。
- 布尔表达式操作符数量。
- 重复判断片段。

推荐重构：

- Extract Method。
- Introduce Explaining Variable。
- Guard Clauses。

自动化等级：

- 简单布尔表达式命名可自动。
- 改控制流结构默认中风险，需要确认。

#### 无用代码

检测依据：

- 优先使用 Symbol Solver 判断 private 方法的 resolved signature 是否被方法调用引用。
- 私有方法/字段无引用。
- 不可达代码。
- 静态分析工具输出。

推荐重构：

- Remove Dead Code。

自动化等级：

- private 且无引用可自动。
- public/protected、反射框架、Spring 注入、序列化字段默认高风险，不自动删除。

#### Feature Envy

检测依据：

- 基于 Symbol Solver 统计方法内部解析成功的方法调用和字段访问。
- 当方法访问外部类型成员明显多于本类成员，并且集中依赖某一个外部类型时，标记为 Feature Envy。
- 忽略 `java.*` 等标准库调用，避免把普通集合、字符串和系统 API 误判为业务职责偏移。

推荐重构：

- Move Method。
- Extract Method。
- 将外部数据访问封装到目标类型内。

自动化等级：

- MVP 默认不自动应用跨类移动代码，只生成计划和 Review 提示。

#### 命名不清晰

检测依据：

- 变量名过短，如 `tmp`, `data`, `obj`, `x`。
- 方法名缺少动词或含义泛化，如 `handle`, `process`, `doIt`。
- 类名过泛，如 `Manager`, `Helper`, `Util`，需结合上下文。

推荐重构：

- Rename Variable。
- Rename Method。
- Rename Class。

自动化等级：

- 局部变量重命名可自动。
- public API、序列化字段、框架绑定名称默认需要人工 Review。

### 6.5 上下文收集模块

模块：`java_context.py`

上下文收集原则：

- 只读取目标类、目标方法、直接调用方、相关测试和必要接口。
- 不读取 `.env`、密钥、证书、构建产物。
- 不把完整仓库上传给远程模型。
- 对远程 LLM 请求做上下文裁剪和敏感路径过滤。

上下文内容：

- 目标方法完整源码。
- 所属类字段和相关方法签名。
- 调用方列表。
- 对应测试类。
- Maven module 信息。
- 静态分析证据。
- 最近一次 scan 的 issue 信息。

### 6.6 重构计划模块

模块：`planner.py`

输入：

- `RefactorIssue`
- `ProjectProfile`
- `JavaContext`
- 测试覆盖初步结果
- 用户配置

输出：

```python
RefactorPlan:
  task_id: str
  issue_id: str
  goal: str
  refactoring_type: RefactoringType
  files_to_modify: list[str]
  expected_changes: list[str]
  out_of_scope: list[str]
  risk_level: RiskLevel
  risk_reasons: list[str]
  verification_commands: list[str]
  rollback_strategy: str
  coverage_assessment: CoverageAssessment
  requires_user_confirmation: bool
```

计划生成策略：

- 规则引擎先生成基础计划。
- LLM 补充解释、边界条件、风险和人工 Review 点。
- 最终计划必须结构化，可落盘，可用于 `apply`。

计划保存到：

```text
.paicli/refactor-agent/tasks/<task_id>/plan.json
.paicli/refactor-agent/tasks/<task_id>/plan.md
```

### 6.7 用户确认模块

应用补丁前必须展示：

- 重构目标。
- 问题位置。
- 修改文件列表。
- 风险等级。
- 覆盖感知结果。
- 将执行的验证命令。
- 回滚方式。
- 高风险警告。

确认交互：

```text
是否应用该重构？[y/N]
```

高风险或缺少覆盖时：

```text
该重构缺少有效测试覆盖，建议先补充 characterization test。
是否生成候选行为锁定测试？[y/N]
```

### 6.8 补丁生成与应用模块

模块：`patcher.py`, `patch_validator.py`, `llm_assistant.py`, `rewrite_runner.py`, `spoon_runner.py`

补丁来源优先级：

1. OpenRewrite recipe dryRun patch。
2. Spoon 自定义 AST 变换输出。
3. LLM 生成结构化 JSON edit operations。
4. LLM 生成 unified diff。

MVP 推荐策略：

- 类/方法定位、private dead code、Feature Envy、调用影响范围、长方法和复杂度统计：优先基于 JavaParser AST + Symbol Solver 摘要。
- Remove private dead code：MVP 已支持确定性自动 patch。
- 简单 Long Method 的 Extract Method：MVP 已支持保守 accumulator block 抽取，并只新增 private helper。
- LLM patch：MVP 已支持结构化 edit operations，必须继续进入计划文件范围校验、JavaParser AST 校验、验证和回滚链路。
- Rename local variable、复杂 Extract Method：后续优先 Spoon 或结构化 edit。
- 复杂 recipe 或生态迁移：优先 OpenRewrite。
- LLM 不直接重写整个文件，只允许输出 unified diff 或结构化操作。

补丁应用要求：

- 应用前创建 refactor 专用快照。
- 应用前校验 patch 只触及计划文件。
- 应用后调用 JavaParser 重新解析计划内 Java 文件。
- 应用后比较 class 声明集合与非 private 方法/构造器签名，避免 LLM 或规则 patch 意外改变外部可见 API。
- 应用后重新读取文件确认变更存在。
- 生成 `git diff --stat` 和 `git diff -- <files>` 摘要。
- patch 应用失败时自动恢复到应用前状态。

禁止行为：

- 不允许格式化整个项目。
- 不允许修改计划外文件。
- 不允许删除未列入计划的文件。
- 不允许直接覆盖完整文件，除非文件很小且用户明确确认。

### 6.9 快照与回滚模块

模块：`rollback.py`

虽然当前项目已有 `suncli_py.snapshot`，但 refactor-agent 需要更严格的任务级快照元数据。

快照内容：

- 当前 HEAD。
- `git status --porcelain`。
- 目标文件修改前内容。
- 计划文件列表。
- 用户已有未提交修改标记。
- task_id。
- 时间戳。

保存位置：

```text
.paicli/refactor-agent/tasks/<task_id>/snapshot.json
.paicli/refactor-agent/tasks/<task_id>/before/
```

回滚原则：

- 只恢复本次任务修改过的文件。
- 若文件在任务后又被用户手动修改，回滚前必须提示冲突。
- 不使用 `git reset --hard`。
- 不覆盖任务开始前已存在的用户未提交修改。

### 6.10 验证流水线模块

模块：`verifier.py`

默认验证流水线：

1. Patch 范围检查。
2. JavaParser AST patch validation。
3. Git diff 检查。
4. 若本任务生成了行为锁定测试，检查其重构前预验证记录。
5. Maven 编译检查。
6. `mvn test`。
7. 静态分析检查。
8. JaCoCo 覆盖感知。
9. 修改文件重新扫描。
10. 报告生成。

推荐命令：

```bash
mvn -q -DskipTests compile
mvn test
mvn org.jacoco:jacoco-maven-plugin:prepare-agent test org.jacoco:jacoco-maven-plugin:report
mvn pmd:cpd-check
```

实际命令需要根据项目 `pom.xml` 和用户配置调整，不能假设所有项目都预装插件。若项目没有配置 PMD 或 JaCoCo，可使用临时 Maven plugin goal 或降级为提示。

验证结果模型：

```python
VerificationResult:
  status: passed | warning | failed
  commands: list[CommandResult]
  diff_summary: DiffSummary
  static_findings: list[StaticFinding]
  coverage: CoverageAssessment
  characterization_tests: CharacterizationTestPlan | None
  changed_files_rescan: list[RefactorIssue]
  rollback_available: bool
```

判定规则：

- 编译失败、测试失败、patch 应用失败：失败。
- 候选行为锁定测试未在重构前通过却继续进入重构：失败。
- 新增行为锁定测试在重构后失败：失败，并提示可能改变原有行为。
- 测试通过但覆盖不足：警告或高风险阻断，取决于风险等级。
- 静态分析有低风险新问题：警告。
- 所有检查通过且覆盖触达：通过。

### 6.11 测试覆盖感知模块

模块：`coverage.py`, `test_generator.py`

MVP 覆盖感知分三层：

1. 测试类存在性：目标类 `Foo` 是否存在 `FooTest`、`FooTests`、`FooIT`。
2. 测试执行性：相关测试是否在默认 `mvn test` 中运行。
3. 行级覆盖：JaCoCo XML 中目标修改行是否 covered。

覆盖评估模型：

```python
CoverageAssessment:
  has_related_test_class: bool
  related_tests: list[str]
  jacoco_report_found: bool
  changed_lines_total: int
  changed_lines_covered: int
  coverage_ratio: float
  confidence: low | medium | high
  needs_characterization_test: bool
  generated_tests: list[GeneratedTestSummary]
  recommendation: str
```

高风险策略：

- 如果 `risk_level = HIGH` 且无相关测试或覆盖率低，默认不进入自动重构，建议先生成 characterization test。
- 覆盖不足时，系统支持在用户确认后生成候选 characterization test，并先验证这些测试在重构前代码上通过。
- 只有候选行为锁定测试在原始代码上通过，系统才允许将其作为后续重构验证依据。
- 如果候选测试无法在原始代码上通过，系统必须停止自动重构流程，报告失败原因，并提示用户人工调整测试或改为只输出重构计划。

测试自动生成流程：

1. 系统识别目标类、目标方法、直接调用方、已有测试类、测试框架和构造依赖。
2. 系统生成候选测试用例，并为每个测试说明锁定的既有行为，例如典型输入输出、边界输入、异常分支、关键状态变化或历史缺陷场景。
3. 系统展示候选测试 diff、测试文件路径、断言意图、mock/fixture 使用方式和潜在风险。
4. 用户确认后，系统优先将测试补充到已有测试类；确实不存在合适测试类时，才新建对应测试类。
5. 系统在重构前运行 `mvn test` 或更精确的相关测试命令，确认新增测试在原始代码上通过。
6. 预验证通过后，系统记录测试基线，并把新增测试纳入后续 `apply` 和 `verify` 的验证命令。
7. 重构完成后，系统再次运行 `mvn test`；如果新增测试失败，应判定为可能改变原有行为，并提示回滚或人工处理。

测试自动生成约束：

- 不得为了让测试通过而修改业务代码。
- 不得删除、跳过或弱化已有测试断言。
- 优先补充已有测试类，避免无计划地创建大量测试文件。
- 优先生成目标方法的单元测试，必要时补充调用链测试。
- 对数据库、网络、文件系统、时间、随机数等外部依赖，应优先使用 mock、fixture 或明确提示无法安全自动生成。
- 如果目标代码副作用过强、依赖过复杂、缺少可构造输入，系统只能生成测试建议，不应强行落地测试文件。
- 候选测试必须只锁定重构前已有行为，不应引入新的业务需求或改变语义预期。

行为锁定测试生成模型：

```python
CharacterizationTestPlan:
  issue_id: str
  target_class: str
  target_methods: list[str]
  test_framework: str
  destination_file: str
  candidate_cases: list[CandidateTestCase]
  requires_new_test_file: bool
  external_dependency_risks: list[str]
  user_confirmed: bool
  pre_refactor_test_result: CommandResult | None
  usable_as_refactor_guard: bool
```

推荐新增命令：

```bash
refactor-agent characterize --issue <id>
```

该命令也可以由 `apply` 流程自动触发：当计划判断覆盖不足时，`apply` 先进入候选测试生成与预验证阶段；预验证通过后再继续执行重构补丁。

### 6.12 报告模块

模块：`report.py`

报告分两类：

1. 扫描报告：坏味道列表、风险分布、建议重构优先级。
2. 重构报告：计划、确认、候选行为锁定测试、重构前测试预验证、快照、diff、验证、覆盖、回滚、结论。

输出位置：

```text
.paicli/refactor-agent/reports/latest.md
.paicli/refactor-agent/tasks/<task_id>/report.md
```

CLI `report` 命令默认展示最新报告摘要。

## 7. LLM 接入设计

### 7.1 支持范围

MVP 明确支持：

- OpenAI。
- Claude。
- DeepSeek。

当前项目已有多 provider LLM 抽象和 DeepSeek 支持。需要新增或完善：

- OpenAI provider。
- Anthropic Claude provider。
- OpenAI-compatible provider 配置能力。

配置示例：

```ini
REFACTOR_AGENT_PROVIDER=openai
OPENAI_API_KEY=...
OPENAI_MODEL=gpt-5

ANTHROPIC_API_KEY=...
ANTHROPIC_MODEL=claude-sonnet-4.6

DEEPSEEK_API_KEY=...
DEEPSEEK_MODEL=deepseek-reasoner
```

### 7.2 LLM 使用边界

LLM 负责：

- 对 JavaParser AST 指标、代码片段、相关测试和调用上下文进行坏味道解释。
- 在规则 fallback plan 基础上补充目标、风险、验证策略和 out_of_scope，生成结构化 RefactorPlan 增强结果。
- 判断风险说明和人工 Review 点。
- 生成受控 JSON edit operations，作为候选 patch 输入。
- 总结验证结果。

LLM 不负责：

- 单独决定是否写文件。
- 绕过用户确认。
- 执行危险命令。
- 大规模重写整个文件。
- 读取或上传完整仓库。
- 直接绕过 patch 范围校验、JavaParser AST 校验、Maven 验证或回滚保护。

### 7.3 Prompt 约束

补丁 prompt 必须要求：

- 只修改计划列出的文件。
- 只处理一个 issue。
- 不做格式化和无关优化。
- 优先输出 JSON edit operations：`file_path`、`start_line`、`end_line`、`replacement`。
- 每个 edit operation 必须是可审查的小范围修改。
- 明确说明不可证明的部分。
- 如果信息不足，返回空 `edits` 或 `NEED_MORE_CONTEXT`。
- 不允许声明 patch 已经安全；安全性由 AST 校验、编译、测试和回滚机制判断。

## 9. 安全设计

### 9.1 路径安全

- 默认 project root 为当前工作目录。
- 所有读写路径必须 resolve 后位于 project root 内。
- 忽略 `.git`、`.paicli`、`target`、`build`、`.gradle`、`node_modules` 等目录。
- 默认不读取 `.env`、密钥、证书、私钥、token 文件。

### 9.2 命令安全

允许命令白名单：

- `git status`
- `git diff`
- `git rev-parse`
- `mvn -v`
- `mvn test`
- `mvn compile`
- `mvn pmd:cpd-check`
- `mvn jacoco:*`
- `mvn rewrite:*`

禁止：

- `git reset --hard`
- `git checkout -- .`
- 删除全目录命令。
- 管道执行远程脚本。
- 提权命令。
- 修改仓库外路径。

### 9.3 模型安全

- 远程 LLM 默认只接收目标上下文，不上传完整仓库。
- prompt 中明确禁止泄露敏感信息。
- 请求日志默认不保存完整源码；若 debug 模式保存，需要脱敏并提示用户。

## 10. 风险分级策略

### 10.1 低风险

典型任务：

- 局部变量重命名。
- private 未使用方法删除。
- 小范围 Extract Method。
- 简单条件表达式解释变量提取。

要求：

- 用户确认。
- 快照。
- diff。
- `mvn test`。

### 10.2 中风险

典型任务：

- public 方法内部 Extract Method。
- 多处重复逻辑抽取。
- 条件结构调整。

要求：

- 用户确认。
- 相关测试存在或提示覆盖不足。
- 编译 + 测试 + 静态分析。
- 覆盖感知。

### 10.3 高风险

典型任务：

- Extract Class。
- Introduce Parameter Object。
- public API Rename。
- 跨类/跨模块共享逻辑抽取。
- 框架反射、Spring Bean、序列化字段相关修改。

要求：

- 强确认。
- 若缺少覆盖，默认建议先补 characterization test。
- MVP 可只输出计划，不自动应用。

## 11. 关键业务流程

### 11.1 scan 流程

```text
读取当前目录
  -> 项目检测
  -> Git 状态检测
  -> Java 文件收集
  -> JavaParser AST + Symbol Solver 解析
  -> AST/符号指标 + 本地规则扫描
  -> 调用 PMD CPD
  -> 汇总坏味道
  -> LLM 基于 AST 指标、代码片段和调用上下文补充解释、风险和建议
  -> 保存 issues.json
  -> 输出扫描摘要
```

### 11.2 plan 流程

```text
读取 issue
  -> 收集目标上下文
  -> 检测相关测试
  -> 初步覆盖评估
  -> 规则引擎生成基础计划
  -> LLM 完善计划、风险和验证方式
  -> 保存 plan.json / plan.md
  -> 输出给用户确认
```

### 11.3 apply 流程

```text
读取 plan
  -> 展示计划与风险
  -> 用户确认
  -> 如果覆盖不足，生成候选行为锁定测试
  -> 展示候选测试 diff、断言意图和风险
  -> 用户确认后写入测试
  -> 在重构前原始代码上运行测试预验证
  -> 预验证通过后记录测试基线
  -> 创建快照
  -> 规则 patch 或 LLM JSON edit operations 生成 patch
  -> 校验 patch 范围
  -> 应用 patch
  -> JavaParser 重新解析并检查 class / 非 private 签名稳定性
  -> 重新读取文件确认
  -> 展示 diff 摘要
  -> 执行 verify
  -> 生成报告
  -> 若失败提示 rollback
```

如果候选行为锁定测试在重构前无法通过，则 `apply` 不得继续应用重构 patch。此时系统应保留测试生成报告和失败输出，并询问用户是否回滚候选测试文件、人工调整测试或只保留重构计划。

### 11.4 rollback 流程

```text
读取 task snapshot
  -> 检查任务后是否存在用户新修改
  -> 若冲突则提示
  -> 恢复任务前目标文件内容
  -> 记录 rollback 结果
  -> 输出恢复摘要
```

### 11.5 characterize 流程

```text
读取 issue 与 plan
  -> 收集目标类、目标方法、调用方和已有测试
  -> 判断测试框架与依赖构造方式
  -> 生成候选行为锁定测试
  -> 展示候选测试 diff、断言意图和风险
  -> 用户确认
  -> 写入已有测试类或新建对应测试类
  -> 在原始代码上运行 mvn test 或相关测试命令
  -> 通过则保存 characterization_precheck.json
  -> 失败则标记测试不可作为重构验证依据
```

## 12. MVP 实施计划

### 阶段一：命令入口与项目检测

目标：

- 新增 `refactor-agent` 命令。
- 实现 `scan` 基础框架。
- 检测 Git/Maven/Java/Test 目录。
- 输出项目 profile。

验收：

- 在真实 Maven Git 仓库运行。
- 能识别非 Maven 项目并给出明确错误。
- 工作区不干净时能提示。

### 阶段二：坏味道扫描

目标：

- 集成 JavaParser helper，优先使用真实 Java AST 提取类、方法、构造器、修饰符和源码范围。
- 集成 JavaParser Symbol Solver，提取方法调用、字段访问、声明类型和 resolved signature。
- 实现长方法、过大类、复杂条件、命名不清晰的本地规则。
- 集成 PMD CPD 做重复代码。
- 初步识别 private dead code。
- 接入 LLM 坏味道解释：输入 AST 指标、代码片段、相关测试和直接调用方；输出影响、风险说明、建议重构方式。
- 输出结构化 issues。

验收：

- 能识别至少 7 类坏味道，包含 Feature Envy。
- 每个 issue 包含类型、文件、位置、严重程度、影响说明、建议方式、是否适合自动重构。
- 对类、方法、构造器、private 成员、方法调用、字段访问和类型归属的识别优先来自 JavaParser AST + Symbol Solver；JavaParser 不可用时应明确提示已降级。
- 配置了 LLM 时，issues 中应包含模型补充的解释和风险 notes；未配置 LLM 时仍可使用规则结果。

### 阶段三：计划生成

目标：

- 实现 `plan --issue`。
- 收集上下文和测试类。
- 规则先生成基础计划。
- 接入 LLM 增强结构化 RefactorPlan，补充目标、风险、验证策略、out_of_scope。
- 保存 plan。

验收：

- 能为指定 issue 生成小步计划。
- 明确修改范围、风险、验证命令、回滚方式。
- LLM 增强计划必须保留规则计算出的文件边界，不能扩大修改范围。

### 阶段四：补丁应用与确认

目标：

- 实现 `apply --issue`。
- 应用前用户确认。
- 创建任务快照。
- 支持低风险任务 patch 生成与应用。
- 支持 LLM 结构化 edit operations 作为候选 patch 输入。
- 支持保守 Extract Method：针对简单 Long Method accumulator block 抽取 private helper。
- 应用后执行 JavaParser AST Patch Validator，检查语法、class 声明和非 private 方法/构造器签名。
- 展示 diff。

验收：

- 用户未确认时不写文件。
- 修改只发生在计划文件内。
- patch 失败不留下半修改状态。
- LLM 越权修改 public API、计划外文件或生成不可解析 Java 时，必须拒绝并恢复任务前内容。
- Extract Method 自动 apply 后应保留原 public 方法签名，只新增 private helper。

### 阶段五：验证与覆盖感知

目标：

- 实现 `verify`。
- 执行 Maven compile/test。
- 集成 JaCoCo 覆盖报告解析。
- 实现覆盖不足时的候选行为锁定测试生成。
- 用户确认后写入候选测试，并在重构前原始代码上验证通过。
- 输出 passed/warning/failed。

验收：

- 测试失败时能显示失败原因。
- 覆盖不足时不把结果误报为完全安全。
- 覆盖不足时，能生成候选测试、展示断言意图，并在用户确认后写入测试文件。
- 新增测试必须先在重构前代码上通过；未通过时不得继续自动重构。

### 阶段六：回滚与报告

目标：

- 实现 `rollback`。
- 实现 `report`。
- 生成完整证据链。
- 报告记录 LLM 是否参与解释、计划增强或 patch 生成，以及 AST Patch Validator 和验证流水线结果。

验收：

- 能恢复本次任务前文件状态。
- 不覆盖任务前已有用户修改。
- 报告包含风险、计划、diff、验证结果、覆盖结果、是否回滚。

## 15. 主要技术风险与应对

### 15.1 Java AST 与 Python 编排之间集成复杂

风险：JavaParser、Spoon/OpenRewrite 是 Java 生态工具，Python 调用需要通过 CLI、Maven plugin 或独立 Java helper；Maven 依赖下载、JDK 版本和 helper 编译失败都可能影响 AST 能力。

应对：MVP 优先通过独立 JavaParser helper 输出 JSON 摘要，并保留文本启发式降级路径；OpenRewrite/Spoon 继续通过 Maven plugin/CLI 调用，减少跨语言 API 绑定。

### 15.2 LLM 生成 patch 不稳定

风险：LLM 可能修改无关代码或输出不可应用 patch。

应对：强制 unified diff 或 JSON edit operations；patch apply 前校验文件范围；失败自动恢复；优先用 OpenRewrite/Spoon 生成可控变更。

### 15.3 覆盖率不能完全证明行为正确

风险：JaCoCo 行覆盖只说明代码被执行，不代表断言有效。

应对：报告中明确“覆盖能证明什么，不能证明什么”；高风险任务缺少有效测试时生成候选 characterization test，并要求新增测试先在重构前代码上通过。

### 15.4 自动生成测试可能锁定错误行为或生成脆弱测试

风险：LLM 可能根据不完整上下文生成错误断言、过度 mock、依赖实现细节，或把偶然行为误当成业务契约。

应对：候选测试必须展示断言意图和覆盖行为；写入前要求用户确认；不得修改业务代码来适配测试；预验证必须在重构前原始代码上通过；报告中标明这些测试只是行为锁定证据，不代表需求正确性。

### 15.5 Maven 项目差异大

风险：多模块、私服依赖、插件配置、JDK 版本会导致命令失败。

应对：项目检测阶段记录 Maven/JDK 信息；验证命令可配置；失败报告给出具体命令、退出码和摘要。

### 15.6 回滚误伤用户修改

风险：任务开始前已有未提交修改，或任务后用户手动编辑同一文件。

应对：任务级 snapshot 记录 before 内容和 git status；rollback 只恢复任务文件；检测任务后新修改冲突并要求确认。

## 16. 不做范围

MVP 不实现：

- Gradle 项目。
- GitHub/GitLab PR Bot。
- IDE 插件。
- Web 平台。
- 大规模架构迁移。
- 跨服务重构。
- 自动无确认修改。
- 高风险重构的默认自动应用。
- 在用户未确认时写入候选测试。
- 为了让候选测试通过而修改业务代码。
- 对副作用强或依赖复杂的代码强行落地测试文件。

## 18. 面试展示建议

建议演示链路：

1. 在一个 Maven 项目中运行 `refactor-agent scan`，展示坏味道列表。
2. 展示 JavaParser Symbol Solver 解析出的调用关系、字段访问和类型归属，说明 Dead Code / Feature Envy 的证据来源。
3. 展示 LLM 基于 AST/符号指标、代码片段、相关测试和调用上下文生成的坏味道解释。
4. 选择一个长方法运行 `refactor-agent plan --issue <id>`，展示规则基础计划和 LLM 增强后的风险、边界、验证策略。
5. 运行 `refactor-agent apply --issue <id>`，展示用户确认、LLM JSON edit operations 或保守 Extract Method patch。
6. 展示修改前快照、AST Patch Validator、patch、diff。
7. 展示 `mvn test` 和覆盖感知。
8. 故意选择一个测试失败样例，展示失败和 rollback。
9. 展示最终报告。

一句话亮点：

> 这个项目不是让 AI 直接大规模改代码，而是把 LLM 放进受控 Coding Agent 流水线里：JavaParser AST + Symbol Solver 负责结构事实、类型解析、调用解析和补丁校验，LLM 负责解释、计划增强和受控 edit operations，最终通过小步补丁、人工确认、Maven 验证、覆盖感知和失败回滚，让自动化重构敢在真实 Java Maven 仓库里落地。
